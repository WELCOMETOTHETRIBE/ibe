# OSCAL Export

IBE can export an assessment as a **documented subset** of NIST OSCAL
(component-definition + assessment-results), mapping model components to selected NIST
SP 800-171 Rev 2 controls with findings derived from the assurance gates.

> **Disclaimer:** This is an OSCAL *subset* for assessment support. It is **NOT** a CMMC
> certification and **NOT** a NIST certification. The exported document even embeds this
> statement in `_ibe_meta.note`: "OSCAL SUBSET generated by IBE. NOT a CMMC/NIST
> certification. For assessment-support use only."

Source: `packages/oscal/export.ts`. Example output:
`evidence/generated/terraform-azure.oscal.json`. CLI: `ibe oscal export`.

## Structure

The exported object has three top-level keys:

- **`_ibe_meta`** — the disclaimer plus explicit `implemented_fields` / `omitted_fields`.
- **`component-definition`** — `uuid: cd-<modelVersion>`, metadata (`oscal-version:
  1.1.2`), and one component per model component, each with a `control-implementations`
  entry (`source: NIST_SP-800-171_rev2`) whose `implemented-requirements` carry a
  `control-id`, a description (`<control title> — <statement>`), and an
  `implementation-status` prop (`implemented | partial | planned`).
- **`assessment-results`** — `uuid: ar-<certificateId>`, one `result` per decision with
  `reviewed-controls`, `observations` (from gate reasons), `findings` (one per gate,
  `status.state: satisfied | not-satisfied`), and `back-matter.resources` referencing each
  evidence id.

Findings map one-to-one to the assurance gates, so an OSCAL reader sees exactly which
governing-rule clause was satisfied or not.

## NIST SP 800-171 controls mapped (MacTech CUI Vault)

`CONTROL_TITLES` in `export.ts` covers nine controls:

| Control | Title |
|---|---|
| 3.1.1 | Limit system access to authorized users, processes, and devices |
| 3.1.3 | Control the flow of CUI in accordance with approved authorizations |
| 3.1.20 | Verify and control connections to external systems |
| 3.3.1 | Create and retain system audit logs and records |
| 3.4.1 | Establish and maintain baseline configurations |
| 3.4.3 | Track, review, approve, and log changes to systems |
| 3.13.1 | Monitor and control communications at external boundaries |
| 3.13.16 | Protect the confidentiality of CUI at rest |
| 3.14.6 | Monitor systems to detect attacks and indicators of potential attacks |

In the terraform-azure demo, the CUI vault component maps to 3.1.3, 3.13.1, 3.13.16, and
3.1.20, each with `implementation-status: partial` and the statement "CUI vault must
remain private; IBE refuses changes that expose it publicly."

## Implemented vs omitted fields

Taken verbatim from `_ibe_meta`:

**Implemented:**
- `component-definition.components`
- `component-definition.components.control-implementations`
- `assessment-results.results.findings`
- `assessment-results.results.observations`
- `assessment-results.results.reviewed-controls`
- `responsible-roles`
- evidence references (via back-matter resources)
- `implementation-status`
- timestamps

**Omitted:**
- SSP (system-security-plan) linkage
- profile / catalog resolution
- parties / locations full metadata
- POA&M (plan of action & milestones)
- digital signatures on the OSCAL doc itself (IBE signs the certificate instead)

## How it ties to the decision

The OSCAL export is generated from the same `GateOutcome[]` the kernel produced and
references the same evidence ids bound into the certificate. It is a compliance-facing
*view* of an assurance decision, not an independent judgement — and the intent can require
`oscal` as a mandatory evidence type, in which case a run without the OSCAL export step
fails the `evidence-complete` gate (as the terraform-azure demo does with
`EVIDENCE_INCOMPLETE — missing: oscal`).
